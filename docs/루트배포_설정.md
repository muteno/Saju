# 루트 배포 설정 — 꼬미 상담을 라이브 주소로 (옵션 A · 260713)

> **왜 필요한가**: 지금 배포(프로젝트 `saju`, 주소 saju-6w4.pages.dev)는 빌드 출력이 **manse 폴더**라 만세 글래스 앱만 서빙된다. **꼬미 상담(루트 사이트 `/`)은 배포에 안 실려서 라이브 주소가 없다**(상담엔진 인계 §12의 1번 미결). 이 문서 = 그 해소법 중 **옵션 A(권장): 루트 출력의 새 Pages 프로젝트 신설** — 기존 주소·만세 앱은 그대로 두고 새 주소를 하나 더 만든다(무파괴).
> **덤**: 루트 프로젝트는 레포 전체를 서빙하므로 만세 앱도 `/manse/app/` 경로로 같이 열린다. `functions/`(서버 계정 API)도 새 프로젝트에 자동 배포 — D1 바인딩만 걸면 가입도 작동.

## 만드는 순서 (전부 클릭 · 단계마다 바로 가는 URL)

1. **새 Pages 프로젝트 만들기** → https://dash.cloudflare.com/b3ca893503b580ac6afba9a7b284d93f/workers-and-pages/create/pages
   **"Git에 연결(Connect to Git)"** → 저장소 `muteno/Saju` 선택 → 다음.
2. **빌드 설정** (여기가 핵심):
   - 프로젝트 이름: `yeul`(추천 — 주소가 yeul-xxx.pages.dev 꼴이 됨) 또는 원하는 이름
   - 프로덕션 브랜치: `main`
   - 프레임워크 프리셋: **None(없음)**
   - 빌드 명령: **비워두기** (빌드 없는 정적 사이트)
   - **빌드 출력 디렉터리: `/`** ← 루트. (기존 프로젝트는 여기가 `manse`였던 것)
   → **저장 후 배포(Save and Deploy)**.
3. **D1 바인딩** (가입 기능까지 열려면) → 배포 끝난 뒤 새 프로젝트 → 설정:
   `https://dash.cloudflare.com/b3ca893503b580ac6afba9a7b284d93f/pages/view/<새프로젝트이름>/settings`
   **바인딩 → 추가 → D1 데이터베이스** → 변수 이름 `DB` · 데이터베이스 `muteno_saju_db` → **프로덕션과 미리보기 둘 다** → 저장 → 재배포 1회.
   (테이블은 이미 만들어져 있으니 스키마 재실행 불필요 — 같은 DB를 두 프로젝트가 같이 쓴다)
4. **확인**: 새 주소 열면 **첫 화면 = 사주 입력 → 꼬미 상담**. `/manse/app/` = 만세 글래스 앱(가입 시트 포함).

## 두 프로젝트 역할 정리 (헷갈림 방지)

| 프로젝트 | 출력 | 주소 | 서빙 내용 |
|---|---|---|---|
| `saju` (기존) | `manse` | saju-6w4.pages.dev | 만세 글래스 앱(= `/app/`) — 그대로 유지 |
| 신설 (옵션 A) | `/` (루트) | 생성 시 표시 | **꼬미 상담(`/`)** + 만세 앱(`/manse/app/`) + 서버 API(`/api/*`) |

- 옵션 B(기존 프로젝트의 출력만 루트로 변경)는 주소 하나로 다 되지만 만세 앱 경로가 `/app/`→`/manse/app/`으로 바뀌는 **파괴적 변경**이라 별도 승인 전 실행 금지(운영자 결정 대기 상태 — 260713).

## 확인 포인트 (만든 뒤 1분)

- 새 주소 `/` → 생일 입력 → "이야기로 풀어줄게" → **꼬미 말투**("왔네. 앉아 봐—")로 상담이 흐르면 성공.
- 같은 기기에서 상담을 끝까지 한 번 보고 **새로고침 후 다시 상담** → 첫마디가 **"또 왔어? …흥, 자리는 비워뒀어."**로 바뀌면 관계 침전물(012)까지 작동 확인.
