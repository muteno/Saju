# 아이콘·픽토그램 정렬 기틀 (이식팩)

> 이 문서는 **어느 레포에나 그대로 이식 가능한 일반형 규칙**이다. UI에서 아이콘 버튼들을 세로/가로 일직선으로 맞출 때 반복해서 어긋나는 문제의 근본을 잡는다. 노뮤트 특정 셀렉터는 「실측 사례」에만 두고, 원칙·검증·재발방지는 전부 일반형으로 서술했다.
>
> 도입 배경: 운영자 관찰 — **디자인 요청의 약 5%가 "이거 정렬 틀어졌으니 잡아달라"**였고, **고쳐도 다른 작업 하다가 또 어긋나** 3일마다 반복. 개별 수정이 아니라 규칙을 기틀에 박아 재발을 없앤다.
>
> **[Yeul 이식 메모 · 260712]** 노뮤트에서 확립된 이 기틀을 §8 절차로 Yeul에 이식. §0~§6·§8 = 포터블 원문 유지, **§7만 Yeul 실측 베이스라인으로 교체**. 요지: Yeul은 아이콘 버튼을 전부 `place-items:center` 셀에 넣어 **현재 정렬 결함이 없다**(실측 확인). 이 문서는 "커질 때"를 대비한 예방 가드레일이다.

---

## 0. 한 줄 요지

**아이콘 버튼들의 세로/가로 정렬 기준은 「버튼(셀 박스)」이 아니라 「버튼 안 픽토그램(아이콘 path)의 시각중심」이다. 그리고 정렬을 지배하는 CSS 값들(컨테이너·헤더·행의 padding, 셀 폭)은 서로 얽힌 연쇄이므로 하나만 바꾸면 반드시 전부 재검증한다.**

---

## 1. 문제 정의 (무엇이 어긋나는가)

- **툴바·팝업·리스트·헤더**에서 아이콘 버튼(닫기 X, 휴지통, 이동 화살표, 공유, 재시도 등)을 세로 또는 가로로 일직선 맞출 때, **셀(버튼 박스) 기준으로 맞추면 눈에는 어긋나 보인다.**
- 대표 사례: **팝업 헤더의 닫기(X) ↔ 각 행의 액션 버튼(휴지통·이동)** 세로 일직선.
- **반복 재발**: 한 번 맞춰도, 이후 다른 작업(패딩 조정·셀 폭 변경·아이콘 교체)을 하면서 정렬이 다시 깨진다. "맞출 때마다 건드려서 또 달라진다."

---

## 2. 근본 원인 (왜 자꾸 어긋나는가) — 4가지

### 2-1. 정렬은 「단일 값」이 아니라 「여러 값의 연쇄 계산」이다
아이콘의 최종 화면 위치 x는 아래의 **합/차**로 결정된다:

```
아이콘 중심 x = 컨테이너 우변
              − 컨테이너 우측 padding
              − 행(또는 헤더)의 우측 padding
              − 셀 폭 ÷ 2
              (± 아이콘 자체 편심)
```

헤더와 행이 **각각 다른 padding·margin 체계**를 쓰면, 이 계산의 어느 한 항만 달라도 아이콘 중심이 어긋난다. **한 값(예: 행 우측 padding 3→0)만 바꾸고 짝이 되는 값(헤더 우측 padding)을 옛 기준 그대로 두면** 즉시 어긋난다. → 이게 재발의 핵심 메커니즘.

### 2-2. 셀 박스 ≠ 눈에 보이는 아이콘
- 버튼 셀은 hover 배경·padding·히트영역(`::before inset`) 때문에 **시각 경계가 아이콘과 다르다.**
- 사용자 눈은 **아이콘(픽토그램)**을 본다. 셀 중심을 맞춰도 아이콘이 셀 안에서 편심돼 있으면 어긋나 보인다.

### 2-3. 방향성 아이콘의 viewBox 편심
- 대칭 아이콘(X, 휴지통, +)은 path 시각중심 = viewBox 중심.
- **방향성 아이콘(↗ 이동, ↻ 재시도, 재생 ▶ 등)은 path가 viewBox 중심에서 편심**될 수 있다. 셀 중앙에 배치해도 아이콘이 한쪽으로 쏠린다.
- (관련: 표지판성 도형은 유니코드 글리프 대신 SVG path로 — 글리프는 폰트 메트릭 때문에 박스 정중앙에서 편심된다.)

### 2-4. 정렬 값이 SSOT로 안 묶여 개별 하드코딩
- 헤더 padding, 행 padding, 셀 폭이 **각각 다른 목적**(제목 여백, 행 리듬, 히트영역)으로 조정되며, 그때마다 정렬이 부수 피해를 본다.
- 정렬을 지배하는 값이 한 곳에 안 묶여 있어, 누가 어디서 건드려도 정렬이 깨진다.

## 3. 정렬 대원칙 (기틀 — 이대로 박는다)

1. **기준 = 픽토그램(아이콘 path) 시각중심. 버튼 셀 박스 아님.**
   - 정렬을 논할 땐 항상 "아이콘의 눈에 보이는 중심"을 말한다.

2. **단일 우측(또는 좌측) 앵커선.**
   - 세로 일직선으로 맞출 여러 버튼은 **같은 앵커 기준선**(예: 컨테이너 우변에서 R−N)을 공유한다.
   - 헤더 닫기와 행 액션이 세로로 맞아야 하면, **헤더의 우측 기준 = 행의 우측 기준**이 되도록 padding 연쇄를 일치시킨다.

3. **정렬 대상 셀은 동일 폭·동일 정렬.**
   - 헤더 닫기 셀 = 행 액션 셀 = **같은 토큰 폭** + `place-items:center`(또는 flex center).
   - 셀 폭이 같고 우측 앵커가 같으면 셀 중심이 같다.

4. **아이콘 크기가 달라도 중심은 일치한다 — `place-items:center`면.**
   - X 15px, 휴지통 16px처럼 크기가 달라도 셀 중앙 배치면 중심 동일. **크기를 굳이 통일할 필요 없다**(시각 균형상 다르게 두는 게 맞을 때도 있다).

5. **편심 아이콘은 보정한다.**
   - 방향성 아이콘은 `getBBox`로 시각중심 편심을 측정해 `transform:translate(±Npx)`로 셀 중앙에 오게 하거나, **대칭 viewBox로 path를 재정의**한다.

6. **정렬 값은 연쇄 SSOT.**
   - 컨테이너 padding · 헤더 padding · 행 padding · 셀 폭을 **한 묶음으로 관리**하고, 하나를 바꾸면 나머지를 함께 조정 + 재검증한다. 독립 변경 금지.

---

## 4. 구현 체크리스트

- [ ] 세로/가로로 맞출 버튼들이 **동일 클래스/토큰의 셀 폭**을 쓰는가
- [ ] **헤더 우측 여백 + 컨테이너 우측 padding = 행 우측 기준**과 일치하는가 (연쇄 계산 확인)
- [ ] 아이콘이 셀 중앙(`place-items:center` / flex center)인가
- [ ] **방향성 아이콘 편심 보정**했는가 (없으면 대칭 path로)
- [ ] 정렬 값을 **SSOT(토큰/상수)**로 묶었는가, 개별 하드코딩 없는가
- [ ] 변경 후 **실측 게이트**(§5) 통과했는가

## 5. 검증 방법 (필수 — 눈짐작 금지)

정렬은 **반드시 픽셀 실측**한다. "맞아 보인다"는 서브픽셀에서 틀린다.

### 5-1. 측정 대상 = svg 요소 박스가 아니라 **path의 시각중심**
`getBoundingClientRect()`(svg 요소 박스 중심)가 아니라, **`getBBox()`(path 실제 렌더 범위)를 절대좌표로 환산한 시각중심**을 잰다. 편심 아이콘은 이 둘이 다르다.

```js
// 아이콘 path의 화면상 시각중심 x (viewBox → 절대좌표)
function pathCenterX(svg) {
  const bb = svg.getBBox();                       // path 범위 (viewBox 좌표)
  const r  = svg.getBoundingClientRect();         // svg 요소 절대 위치
  const vb = svg.viewBox.baseVal;                 // 예: 0 0 24 24
  const scaleX = r.width / (vb.width || 24);
  return r.left + ((bb.x + bb.width / 2) - (vb.x || 0)) * scaleX;
}
```

### 5-2. 합격 기준
- 세로 정렬: 맞출 아이콘들의 `pathCenterX`가 **±1px 이내**(0 지향).
- 가로 정렬: `pathCenterY`(위 식의 y축 버전)가 ±1px 이내.
- **서브픽셀(0.2~0.5px) 잔여는 브라우저의 홀짝 픽셀 반올림 때문에 불가피**할 수 있다. 1px을 넘으면 원인(§2)을 찾아 잡는다.

### 5-3. 헤드리스 렌더로 좌표 실측 (예: Playwright)
- 실제 팝업/툴바를 렌더 → 각 아이콘 `pathCenterX` 비교 → 오차 리포트.
- 스크린샷만으로 판단하지 말 것(스크린샷도 서브픽셀은 못 본다). **좌표 숫자로** 확인.
- **Yeul 실측 도구**: 이 레포엔 playwright npm이 없다. 대신 `/opt/pw-browsers/chromium` 바이너리를 헤드리스로 직접 구동 — 정렬 관련 CSS·SVG를 그대로 옮긴 자기완결 하네스(`file://`, 서버·백그라운드 프로세스 0)를 `--dump-dom`으로 좌표 뽑고 `--screenshot`으로 시각 확인. (§7에서 이 방식으로 베이스라인을 쟀다.)

## 6. 재발 방지 (가장 중요)

1. **정렬 값 연쇄 규칙**: 컨테이너/헤더/행 padding, 셀 폭 중 **하나라도 바꾸면 정렬 실측 재검증**을 커밋 전 필수로 건다.
2. **SSOT화**: 정렬을 지배하는 값은 토큰/상수 한 곳에. "이 팝업만" 하드코딩 금지.
3. **자동 게이트(가능하면)**: CI/훅에서 대표 팝업의 아이콘 정렬을 헤드리스로 측정해 1px 초과면 실패시킨다. 사람 눈에 의존하지 않는다.
4. **아무 UI 작업이든** 정렬 대상 근처의 padding/폭/아이콘을 건드리면 이 문서의 체크리스트를 돈다.

## 7. 실측 사례 (Yeul — 260712 베이스라인)

> 원본 이식팩의 §7은 노뮤트(no_mute) 사례였다(헤더형 팝업 3종의 닫기X ↔ 행 액션 세로 일직선, 서브픽셀 잔여를 §2-1 짝 값 미갱신으로 규명). Yeul엔 그 "리스트 팝업" 패턴이 아직 없어, 아래는 **Yeul 현행 아이콘 전수 실측**으로 대체한다.

- **대상(아이콘 버튼/픽토그램 전수)**: 상단바 `.icon-btn` 2종(테마 토글 🌙 · 설치 ⤓), 모달 헤더 닫기 `.icon-btn`(X), 장면 카드 우상단 `.scene-open`(↗ `arrowUpRight`), 히어로 CTA `.btn`(→ `arrow` · ▶ `play`), pane 헤더 `.pill-btn`(복사·이미지), 푸터 `.socials a`(X·IG·GH), 우하단 `.fab`(↑), 관객 카드 `.aud-ico`.
- **구조**: 정렬 대상이 되는 **독립 아이콘 버튼은 전부 `display:grid; place-items:center`**(또는 flex center) 셀에 들어간다 — `.icon-btn`(42px) · `.socials a`(34px) · `.fab`(52px) · `.aud-ico`(56px). 즉 §3-3·§3-4를 **이미 충족**. 정렬 지배값은 `css/styles.css`의 컴포넌트 클래스에 있고, 셀 폭·radius·간격은 `:root` 토큰(`--pill` 등) 또는 컴포넌트 로컬 상수로 관리.
- **⚠ 유일한 실질 발견 — `.scene-open`(↗)은 자기 박스가 없다(§2-4·§3-4 미충족)**: 다른 아이콘 버튼과 달리 `.scene-open`엔 `width/height/display/place-items`가 **전혀 없다**(`styles.css:246`은 `margin-left:auto` + 색·전이만). 지금 ↗ 우변 열이 맞는 건 오직 ① `ICON.arrowUpRight`가 **인라인 `width/height="18"`**을 실어 오고(`005-ui-kit.js:10`) ② 세 카드가 **하드코딩 26px 패딩을 공유**하기 때문 — **우연에 기댄 정렬**이다. 크기 미지정 SVG(예: `AUDIENCE_ICON` 글리프들은 인라인 크기가 없다)를 여기 끼우면 `.scene-open svg`를 제약하는 CSS가 없어 **레이아웃이 터진다.** 현재는 정상이지만 **레포에서 가장 회귀 취약한 지점.** → 권고: `.scene-open`에 다른 아이콘 버튼처럼 `display:inline-grid; place-items:center`(+ `line-height:0` 또는 고정 셀)로 **중앙 정렬을 셀에 강제**해 우연 의존을 없앤다. 시각상 no-op(↗ 위치 불변). *운영자 승인 후 적용 — 이 이식 커밋은 감사·기록까지만.*

  - `play`(▶ `M8 5v14l11-7z`) = 삼각형 bbox x[8,19] 중심 13.5 → **+1.5단위(화면 ≈ +1.0px 우측)**. 단 "▶ 예고편"처럼 **글자 이끄는 단독 아이콘**이라 열 정렬 대상 아니고, 재생 아이콘 광학 무게중심 범위 안이라 **현행 유지**. 여러 ▶를 정렬하는 UI 생기면 그때 §3-5 보정.
  - `draft`(관객 카드 문서 아이콘) = 본문 x[4,19] 중심 11.5 → **−0.5단위(화면 ≈ −0.56px)**, 경계선·서브지각. 단독·비정렬 대상 → 유지.
  - 그 외(설치·테마·소셜·fab·battery·exit·film 등) = **편심 0**.
- **결론**: **Yeul은 현재 아이콘 정렬 결함이 없다.** 없는 문제를 만들어 고치지 않는다(§CLAUDE "과설계 경계"). 이 문서는 **커질 때를 대비한 예방 가드레일** — 새 툴바·리스트·팝업(헤더X ↔ 행 액션 세로열 등)을 만들 때 §3~§5를 돌려 재발을 원천 차단한다.
- **교훈(원본 §7 계승)**: 헤더 padding과 행 padding은 짝이다. 한쪽을 바꾸면 다른 쪽을 같이 재계산해야 한다(§2-1). Yeul에 리스트 팝업이 생기는 순간 이 교훈이 바로 발동한다.
