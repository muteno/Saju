> **용도**: 프로필 체계(사주·에니어그램·MBTI·타로·별자리 등)를 "진짜 상담받는 맛"의 대화로 푸는 앱이라면 어디든 이식하는 공통 팁팩. **이 문서 하나만 참조해도 돌게** 자기완결로 씀.
> **관계**: `docs/상담엔진_설계.md` = 이 팩을 사주(만세력)에 인스턴스화한 정본. 이 팩 = 체계 불문 일반형.
> **출처**: 사주 유튜브 자막 308편(4채널 교차)·실전 상담사례 76편·정곡 랭킹·고리 상태기계 분석의 증류물. 원리는 체계를 가리지 않는다.

## §0. 전제 — 이식 가능한 앱의 조건

1. **결정론적 계산기**가 있다: 입력(생일·검사 응답 등) → 프로필(특징 집합)을 재현 가능하게 산출. `<계산기>`
2. 프로필에서 **참/거짓 판정 가능한 특징**들을 뽑을 수 있다(예: "金 0개", "유형 5w4", "타워 카드"). `<특징셋>`
3. 대화형 UI(버튼 선택지 가능). — 이 세 개가 없으면 이 팩은 붙을 자리가 없다.

## §1. 핵심 공식 — 프로필 = 프롬프트, 상담 = 자기회귀 트리 샘플링

| LLM | 상담 엔진 |
|---|---|
| 프롬프트 | `<계산기>` 산출 프로필 — 사전 정보 0에서 시작 |
| 어휘 | 정곡 후보 풀(**계산이 참인 특징만**) |
| 로짓 | 임팩트 점수(§2) |
| 다음 토큰 샘플링 | 다음 고리 선택 |
| 컨텍스트 갱신 | 사용자 답이 상태에 들어가 다음 선택 재가중 |
| 가지치기/부스트 | "아니야"=그 계열 prune / "맞아"=인접 후보 boost |
| 온도 | 템포 규칙(§7) + 적중 누적(trust) 어닐링 |
| 디코더 | 표현층 — 문구(slot)와 근거(조건 술어)를 **반드시 분리**(후일 LLM 스왑 가능) |

핵심 성질 = **자기회귀**: 고정 대본이 아니라 매 답이 다음 선택 분포를 바꾸는 샘플러. 이래야 "살아있는 상담" 느낌이 난다.

## §2. 정곡 선별 — 뭘 먼저 쏠까

- **후보 자격 = 계산이 참인 것만.** 없는 특징은 언급 자체 금지(콜드리딩과의 첫 경계선).
- **임팩트 = 희소성×0.35 + 체감성×0.40 + 안전성×0.25.** 체감성 서열: 성향 < 사건 < **시기 특정**("○살 무렵" — 최강).
- **단계당 대표 정곡 1발**(몰아 쏘지 않기) + 큰 정곡 직전엔 연출로 "계산이 진짜다" 각인.
- **밋밋한 프로필 폴백**: 특징을 창작하지 않는다. 항상 존재하는 구조적 사실(중심 유형·최다 축·현재 전환기)로 골격 투어 + "균형 자체가 특징" 정직 프레이밍.

## §3. 고리 체인 — 파고드는 구조

```
고리 = 훅(계산 사실 한 방) → 전개(뜻풀이) → 확인(버튼) → 분기 → 다리(다음 예고)
```
- **세로 고리**(백미): 한 특징이 전 단계를 관통 — 결핍 발견 → 그게 못 하는 일 → 삶의 영역 → 처방이 그걸 보충 → 그게 풀리는 시기. 하나의 실이 끝까지.
- **가로 고리**: 같은 단계 안 옆 확장. 세로 우선, 가로는 "맞아" 뒤 한 뼘만.
- **드릴다운 사슬 공식**(실전 상담 실증): 특징 → 성향 → 삶 영역(일·연애·돈·건강) → 시기 → 조언.
- **"아니야" 처리 = 이 팩의 급소**: 그 계열 통째 prune + 정직 보정 한 줄 + 다음 후보로 **우아하게**. "사실 그것도 맞는 거야" 식 재해석(무오류 우기기) = 즉시 콜드리딩 전락. 틀림을 인정하는 게 신뢰를 산다.
- **"맞아" 처리**: 파고들기 1~2줄(재확인 없이) + 인접 후보 boost.

## §4. 답변지 — 맞아/아니야를 넘어서

정곡이 꽂힌 순간("억?") **계산된 후보 답안 3~4개 + [아니야]를 버튼으로**. 보기 자체가 계산 산물이어야 한다(예: 충 맞은 궁이 관장하는 영역 → [이직] [이사] [관계 정리]). 어떤 보기든 후속 가지가 사전 준비 → "다 알고 있었네" 체감 배가. = **top-k 제약 디코딩**(자유 입력 대신 계산된 보기).

> D3 실구현 규칙(사주 인스턴스 검증됨): EVENT 정곡에만 답변지 적용 · 보기 = 그 궁(자리)이 관장하는 삶 영역(궁위론)에서 2개 + 일반 장면 1개 · 궁 미특정(전환기 등) = 일반 3보기 · 마지막 버튼 = "그런 일 없었어"(=prune 경로) · 보기 적중 시 고른 장면을 받아쳐 파고든다("그래 — {보기}. 그 자리가 맡는 일이 그렇게 온 거야").

## §5. 정직 3단 어조 — 리딩과 콜드리딩의 경계선

| 층 | 내용 | 어조 |
|---|---|---|
| CALC | 계산 사실 | **단정 OK** — "네 프로필에 X가 있어" |
| INFER | 통설 해석 | **hedge 강제** — "~한 경향이 있을 수 있어" (+참고 라벨) |
| EVENT | 과거·미래 사건 | **질문형만** — "그 무렵 ~있지 않았어?" (단정 금지) |

금지선(장르 공통, 원조 화자들도 명시 거부): 사망·이혼·합격 시점 단정 · 공포 마케팅 · 결핍의 부정 서사(반드시 보완법으로 연결) · 좋다/나쁘다 단면 렌더(양면 병기) · 세일즈 톤.

## §6. 목소리 공통분모 — 한국어 리딩 화법 10결 (4화자 교차 실측)

①**가능성 화법이 기본값**("~볼 수도 있어" — 코퍼스 881회) ②**흉→활용 전환을 항상 세트로**("활용" 489회) ③구조론(운명론 아님 — "정해진 건 없다") ④위로 한 스푼("네 탓 아니야") ⑤전문어 1개당 일상 비유 1개 즉시 번역 ⑥**"결국 선택은 너" 주체성 클로징** ⑦미신·겁주기 비판 ⑧이분법 거부 ⑨실행 조언으로 끝맺기 ⑩따뜻한 사제 톤. — 엔딩 3종 세트 = **처방 + 시기 + 격려**(판정으로 끝내지 않기).

## §7. 템포

- 확인 버튼 예산: 세로 체인당 **2회**(진입 훅 1 + 큰 정곡 1). 휴리스틱(soft) 진술엔 확인 안 붙임(틀릴 수 있는 걸 "맞지?"로 압박하면 신뢰 하락).
- **3고리마다 큰 정곡 1방** 보장. 단계 전환마다 요약 카드 1장.
- 연출(카드 플립·게이지·타임라인)은 큰 정곡 **직전**에 — 말보다 그림 먼저.

## §8. 겸손한 전지 스탠스

- **백스테이지**: 전체 트리·모든 가지 사전 계산(그래서 어떤 보기를 골라도 즉시 이어짐 — 연기가 아니라 사실).
- **프론트스테이지**: "다 안다" 말투 ✗ → "구조는 이렇게 읽히고, 나머지는 네 몫" ✓. 리딩형 질문("있었지?")은 쓰되, 겸손·유보가 표면 톤(원조 화자 공통 결).
- 빗나가도 안 당황 — 당황하지 않는 태도가 전지함의 구현이지, 우기기가 아니다(§3 급소와 동일).

## §9. 스타일 라우팅 — 유형론의 진짜 쓰임

- 타 유형론(애니어그램 등)의 최고 쓰임 = 유형 '내용'이 아니라 **발화 스타일 선택기**: 같은 사실을 안정형=대비 프레임 / 변화형=전환 프레임 / 분석형=근거 선행 / 직진형=hedge 최소로 포장.
- 청자 프로파일은 대개 **자체 계산기에서 이미 나옴**(추가 검사 불요).
- 사용자가 타 체계 유형을 **직접 제공**하면 → "두 독립 체계가 같은 방향을 가리킨다"는 교차 관찰 = 강한 정곡. 단 관찰로만(융합 주장·1:1 대응표 노출 금지).
- **스타일은 포장만** — 계산 사실을 숨기거나 뒤집지 못한다(§5 우선).

## §10. 이식 절차 + 안티패턴

**절차**: ① `<계산기>` 출력 필드 전수 나열 → ② 각 필드에서 참/거짓 특징 후보 정의(조건 술어) → ③ §2 3축 점수 매김 → ④ 단계 골격 확정(그 체계의 정통 순서를 따르되 특이점 하이라이트 선행) → ⑤ 세로 고리 3~5개 설계 → ⑥ §6 목소리 상수로 문구 작성(slot 분리) → ⑦ 3케이스 검증(특징 많은 프로필 / 밋밋한 프로필 / 정보 불완전 프로필 — 불완전이면 의존 판정 **정직 보류**).

**안티패턴**:
- 특징 창작(밋밋하다고 지어내기) — 전체 신뢰 붕괴의 시작.
- 무오류 우기기("그것도 맞는 거야") — 콜드리딩 전락.
- 스킨 먼저(캐릭터·연출부터 입히기) — 뼈대(계산·정곡·고리) 없이 스킨만 있으면 "예쁜 뻥"이 된다. 스킨은 마지막.
- 정보 불완전을 임의값으로 때우기(예: 모르는 시간→정오) — 파생 판정 전체가 오염된다. 보류가 정답.
- 확인 남발(매 문장 "맞지?") — 피로 + 자신 없어 보임.
- 특정 화자 문장 복붙 — 구조·통설·키워드만 인코딩(저작권+품질 둘 다).

## §11. 페르소나 팩 도킹 스펙 — 실물 인터페이스 (사주 앱 구현 기준 · 260713)

> 다른 드래프트(페르소나·화법 팩)가 이 엔진에 합류할 때 맞춰야 할 접점. 사주 인스턴스의 실제 구현
> (`js/knowledge/008-jeonggok`=선별기 · `009-consult-graph`=진행)에 이미 존재하는 슬롯 기준이라, 여기만 갈아끼우면 스킨이 입혀진다.

**엔진이 페르소나 팩에게 주는 것(입력)** — 후보 객체:
```
{ key, step(1~5), impact(수치), tone:'DIRECT'|'SOFT', label(계산 사실 요약), gloss(중립 한줄), facts{...근거} }
```
- `tone:'SOFT'` = 흉계열(충·원진·형·귀문·백호·공망·혼잡) — **순화(활용형·질문형)가 페르소나 화법의 주 무대.**
- `impact ≥ 3.6` = 큰 정곡 — 연출·강조가 허용되는 지점.

**페르소나 팩이 제공해야 하는 것(출력)** — 스왑 대상 6종:
1. **훅 문구 사전**: 후보 key(패밀리)별 훅 1줄 — 현재 v1 중립판이 `hookText()`에 있음. 같은 키 체계로 교체판 제공.
2. **전개 문구 사전**: 동일 구조(`developText()` 대응). 가능성 화법(hedge) 유지 필수 — 페르소나가 입혀져도 §5 3단 어조는 불변.
3. **확인 프롬프트 + 답변지 보기 문구**: 질문 어미·보기 라벨의 말투(`confirmPrompt()` 대응). EVENT는 질문형 강제.
4. **보정·받아치기 문구**: "아니야" 후 정직 보정(`RECAL` 대응 — 우기기 금지 그대로), "맞아" 후 파고들기 어투.
5. **엔딩 3종 템플릿**: 처방·시기·격려의 페르소나 버전(주체성 클로징 유지).
6. **스타일 라우팅 프리셋**(선택): 청자 유형축 → {hedge 수위, 훅 강도, 비유 계열, 위로:도전 비율} 매핑(§9). 유형 라벨 노출 금지, 내부 전용.

**불변 조항(페르소나가 못 건드리는 것)**: 후보 선별·랭킹·게이팅(참인 것만·보류 규칙) · 3단 어조의 층위 구분 · 금지선(§5) · "아니야"=prune. **페르소나는 어떻게 말하느냐만 바꾼다 — 무엇을 말하느냐는 계산의 몫.**
